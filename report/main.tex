    \section{Introduction}
    \pagenumbering{arabic}

    \subsection{Background}

    Document similarity search is a fundamental problem in information retrieval,
    data mining, and machine learning. With the exponential growth of digital
    content, finding documents most similar to a given query efficiently has become
    critical across applications ranging from plagiarism detection and duplicate
    identification to recommendation systems and semantic search \cite{manning2008}.

    Traditional sequential similarity computations become computationally
    prohibitive at scale: comparing a query against $n$ documents each of
    dimension $m$ costs $O(n \cdot m)$ floating-point operations per query.
    Parallel computing frameworks -- specifically the Message Passing Interface
    (MPI) for inter-node distribution and OpenMP for intra-node thread-level
    parallelism -- provide scalable solutions by distributing computation across
    multiple processors \cite{gropp1999mpi}.

    This project implements a \textit{hybrid parallel} document similarity search
    system combining MPI and OpenMP. The system distributes documents across MPI
    ranks using an optimal collective scatter, parallelises per-document similarity
    computation within each rank using OpenMP threads, and merges per-rank top-$k$
    results via a single MPI collective gather. Four similarity metrics are
    supported and comprehensive per-phase timing is reported.

    \subsection{Problem Statement}

    Given a collection of $n$ documents (each represented as an $m$-dimensional
    integer feature vector) and a query vector $\mathbf{q} \in \mathbb{Z}^m$,
    find the $k$ most similar documents to $\mathbf{q}$ in minimum wall-clock time.

    \begin{itemize}
        \item \textbf{Input}: $n$ documents in a plain-text file, a query vector,
              parameters $k$ and $m$
        \item \textbf{Output}: Top-$k$ documents ranked by similarity score,
              with full four-phase performance statistics
        \item \textbf{Challenge}: Minimise total execution time
              $T_{\text{total}} = T_{\text{I/O}} + T_{\text{scatter}} +
              T_{\text{comp}} + T_{\text{gather}}$
    \end{itemize}

    \subsection{Discussion of Literature}

    Johnson et al.\ \cite{johnson2021faiss} and the subsequent FAISS library
    paper \cite{douze2024faiss} establish GPU-based $k$-selection and product
    quantisation as the current frontier for large-scale similarity search.
    Their WarpSelect algorithm -- a register-resident tournament sort --
    motivates our use of a CPU-side min-heap for $k$-selection, which achieves
    the same $O(n \log k)$ complexity without requiring GPU hardware.

    Malkov and Yashunin \cite{malkov2020hnsw} and Aumuller et al.\
    \cite{aumuller2020ann} together provide context for exact versus approximate
    search. Our implementation performs \textit{exact} search, which is
    appropriate when result correctness is required and dataset size allows it.

    The hybrid parallelism literature \cite{lu2023hybrid,velarde2022hybrid}
    directly motivates our MPI+OpenMP design: on multi-core nodes, OpenMP
    threads share memory and cache lines, avoiding the per-rank data replication
    that pure MPI incurs. Thakur et al.\ \cite{thakur2005mpi} justify replacing
    sequential \texttt{MPI\_Send} scatter loops with \texttt{MPI\_Scatterv},
    which delegates communication scheduling to the MPI runtime's optimal tree.

    \section{Research Gaps}

    Based on the literature review, the following gaps were identified and
    addressed in this project:

    \begin{itemize}
        \item \textit{\textbf{Single-Level Parallelism}}: Most prior CPU-based
              MPI implementations exploit only inter-process parallelism.
              On multi-core nodes, pure MPI wastes resources by replicating data
              per rank. Our hybrid MPI+OpenMP design exploits both levels.

        \item \textit{\textbf{Inefficient Top-K Selection}}: Many implementations
              sort all $n$ local documents ($O(n \log n)$) even when only
              $k \ll n$ results are needed. A min-heap reduces this to
              $O(n \log k)$ with $O(k)$ memory.

        \item \textit{\textbf{Suboptimal Scatter Pattern}}: Sequential
              \texttt{MPI\_Send} loops from rank 0 to each worker incur
              $O(P)$ sequential latencies. Replacing with \texttt{MPI\_Scatterv}
              reduces scatter cost to $O(\log P)$ via the runtime's binomial tree.

        \item \textit{\textbf{Correctness in KReduce}}: When fewer documents
              than $k$ exist on a rank, passing a variable send count to
              \texttt{MPI\_Gather} causes memory corruption.  Uniform sentinel
              padding to exactly $k$ entries per rank fixes this.

        \item \textit{\textbf{Limited Metric Flexibility}}: Most systems support
              only cosine similarity.  Four metrics (Power, Cosine, Euclidean,
              Manhattan) are exposed through a unified interface here.

        \item \textit{\textbf{Coarse Performance Profiling}}: Few implementations
              separate I/O, scatter, computation, and gather phases.
              Our four-phase \texttt{MPI\_Wtime} instrumentation enables
              precise bottleneck identification.
    \end{itemize}

    \section{Problem Formulation}

    \subsection{Mathematical Formulation}

    Let $\mathcal{D} = \{\mathbf{d}_1, \mathbf{d}_2, \ldots, \mathbf{d}_n\}$
    be a collection of $n$ documents where each $\mathbf{d}_i \in \mathbb{Z}^m$.
    Given a query $\mathbf{q} \in \mathbb{Z}^m$, find the set $\mathcal{T}_k$ of
    the $k$ most similar documents:

    \begin{equation}
        \mathcal{T}_k = \underset{\mathcal{S} \subseteq \mathcal{D},\,
        |\mathcal{S}|=k}{\arg\max} \sum_{\mathbf{d} \in \mathcal{S}}
        S(\mathbf{q}, \mathbf{d})
    \end{equation}

    Total execution time is decomposed into four measurable phases:

    \begin{equation}
        T_{\text{total}} = T_{\text{I/O}} + T_{\text{scatter}} +
        T_{\text{comp}} + T_{\text{gather}}
    \end{equation}

    \subsection{Supported Similarity Metrics}

    \begin{enumerate}
        \item \textbf{Power-based Similarity}:
        \begin{equation}
            S_{\text{power}}(\mathbf{q}, \mathbf{d}_i) =
            \sum_{j=1}^{m} d_{i,j}^{q_j}
        \end{equation}

        \item \textbf{Cosine Similarity}:
        \begin{equation}
            S_{\text{cosine}}(\mathbf{q}, \mathbf{d}_i) =
            \frac{\mathbf{q} \cdot \mathbf{d}_i}
                 {\|\mathbf{q}\|_2 \, \|\mathbf{d}_i\|_2}
        \end{equation}

        \item \textbf{Euclidean Distance} (negated for maximisation):
        \begin{equation}
            S_{\text{euclidean}}(\mathbf{q}, \mathbf{d}_i) =
            -\sqrt{\sum_{j=1}^{m} (q_j - d_{i,j})^2}
        \end{equation}

        \item \textbf{Manhattan Distance} (negated for maximisation):
        \begin{equation}
            S_{\text{manhattan}}(\mathbf{q}, \mathbf{d}_i) =
            -\sum_{j=1}^{m} |q_j - d_{i,j}|
        \end{equation}
    \end{enumerate}

    Negating distance metrics ensures a uniform ``higher score = more similar''
    convention across all four metrics.

    \subsection{Parallel Formulation}

    With $P$ MPI ranks and $T$ OpenMP threads per rank (total
    $P \times T$ hardware threads), documents are partitioned using
    \textbf{contiguous block distribution}:

    \begin{equation}
        |\mathcal{D}_r| =
        \left\lfloor \frac{n}{P} \right\rfloor +
        \begin{cases}
            1 & \text{if } r < (n \bmod P) \\
            0 & \text{otherwise}
        \end{cases}
        \quad r = 0, 1, \ldots, P-1
    \end{equation}

    Each rank $r$ computes a local top-$k$ set $\mathcal{T}_k^{(r)}$
    using $T$ OpenMP threads. The global result is:

    \begin{equation}
        \mathcal{T}_k = \text{TopK}\!\left(
        \bigcup_{r=0}^{P-1} \mathcal{T}_k^{(r)}\right)
    \end{equation}

    This block distribution guarantees load imbalance of at most one document
    between any two ranks, regardless of $n$ and $P$.

    \section{Objectives}

    \begin{itemize}
        \item To implement a \textbf{hybrid MPI+OpenMP} parallel document
              similarity search system that exploits both inter-node and
              intra-node parallelism simultaneously.

        \item To optimise top-$k$ selection using a \textbf{min-heap} data
              structure, achieving $O(n \log k)$ time and $O(k)$ space
              per rank instead of $O(n \log n)$ for a full sort.

        \item To replace sequential \texttt{MPI\_Send} scatter loops with
              \textbf{\texttt{MPI\_Scatterv}}, reducing scatter latency from
              $O(P)$ to $O(\log P)$ via the MPI runtime's binomial tree.

        \item To implement a \textbf{correctness-preserving KReduce} that pads
              each rank's local top-$k$ to a uniform $k$ entries with sentinel
              values before \texttt{MPI\_Gather}, preventing memory corruption
              when local document counts fall below $k$.

        \item To support \textbf{four similarity metrics} (Power, Cosine,
              Euclidean, Manhattan) through a unified \texttt{computeSimilarity}
              interface enabling runtime metric selection.

        \item To provide \textbf{four-phase performance profiling}
              (I/O, scatter, computation, gather) using \texttt{MPI\_Wtime}
              and to analyse scalability using Amdahl's and Gustafson's Laws.

        \item To validate correctness by confirming \textbf{identical top-$k$
              rankings} across all processor and thread configurations.
    \end{itemize}

    \section{Methodology}

    \subsection{System Architecture}

    The system follows a master--worker model with two levels of parallelism:

    \begin{itemize}
        \item \textbf{Level 1 -- MPI (inter-process)}: Rank 0 reads all input
              files, broadcasts the query vector and metadata to all ranks, and
              distributes document partitions via \texttt{MPI\_Scatterv}.
              After computation, rank 0 collects per-rank top-$k$ results via
              \texttt{MPI\_Gather} and performs the global merge.

        \item \textbf{Level 2 -- OpenMP (intra-process)}: Within each MPI rank,
              a \texttt{\#pragma omp parallel for} loop with
              \texttt{schedule(dynamic, 64)} distributes similarity computation
              across $T$ threads. The expensive similarity functions
              (\texttt{pow}, \texttt{sqrt}, multiply-accumulate) are fully
              parallelised; the min-heap is maintained sequentially afterward
              to avoid synchronisation overhead.

        \item \textbf{Communication Primitives}:
              \texttt{MPI\_Bcast} for query and metadata;
              \texttt{MPI\_Scatterv} for document distribution (single collective,
              variable per-rank counts);
              \texttt{MPI\_Gather} for top-$k$ result collection (uniform
              $k$ entries per rank via sentinel padding).
    \end{itemize}

    \subsection{Algorithm Design}

    \subsubsection{Min-Heap Based Top-K Selection}

    \begin{algorithm}[H]
        \caption{Min-Heap Top-K Selection (per MPI rank)}
        \begin{algorithmic}[1]
            \State \textbf{Input:} Local documents $\mathcal{D}_r$,
                   query $\mathbf{q}$, integer $k$
            \State \textbf{Output:} Local top-$k$ array \textit{local\_topk}
            \State \Comment{Phase A: parallel score computation (OpenMP)}
            \For{each document $\mathbf{d} \in \mathcal{D}_r$
                 \textbf{in parallel} ($T$ OMP threads)}
                \State $\text{scores}[i] \gets S(\mathbf{q}, \mathbf{d})$
            \EndFor
            \State \Comment{Phase B: sequential heap maintenance}
            \State $H \gets \text{CreateMinHeap}(k)$
            \For{$i = 0$ \textbf{to} $|\mathcal{D}_r| - 1$}
                \State \Call{InsertHeap}{$H$, $\text{id}[i]$, $\text{scores}[i]$}
            \EndFor
            \State \Return \Call{ExtractAllSorted}{$H$}
        \end{algorithmic}
    \end{algorithm}

    Splitting into two phases is essential: the min-heap is not thread-safe.
    By pre-computing all scores in Phase A and inserting sequentially in
    Phase B, we obtain full parallelism on the expensive computation and
    zero synchronisation overhead during heap operations.
    Complexity per rank: $O(n \cdot m / (P \cdot T))$ for Phase A and
    $O((n/P) \log k)$ for Phase B, versus $O(n/P \cdot \log(n/P))$
    for a full sort \cite{cormen2022algorithms}.

    \subsubsection{Robust KReduce with Sentinel Padding}

    \begin{algorithm}[H]
        \caption{KReduce -- Correctness-Preserving Global Top-K}
        \begin{algorithmic}[1]
            \State \textbf{Input:} \textit{local\_topk} (size \textit{actual\_k}
                   $\le k$) from all $P$ ranks
            \State \textbf{Output:} \textit{global\_topk} at rank 0
            \State Pad \textit{local\_topk} to exactly $k$ entries:
            \For{$i = \text{actual\_k}$ \textbf{to} $k - 1$}
                \State $\text{padded}[i] \gets (\text{id}{=}-1,\;
                       \text{score}{=}-\infty)$
            \EndFor
            \State \Call{MPI\_Gather}{\textit{padded}, $k \times
                   \text{sizeof(DocScore)}$, \texttt{MPI\_BYTE}, $\ldots$,
                   root=0}
            \If{rank $= 0$}
                \State Filter entries with $\text{id} = -1$ (sentinels)
                \State \Call{MergeSort}{\textit{valid\_entries}, descending}
                \State $\text{global\_topk} \gets
                       \text{valid\_entries}[0 : k]$
            \EndIf
        \end{algorithmic}
    \end{algorithm}

    The sentinel padding ensures that all $P$ ranks always send exactly $k$
    entries, satisfying \texttt{MPI\_Gather}'s requirement for a uniform send
    count. Without this, any rank with $|\mathcal{D}_r| < k$ would send a
    different byte count, causing memory corruption or a runtime abort.

    \subsubsection{OpenMP Intra-Rank Parallelism}

    Within each MPI rank, the similarity computation loop is parallelised:

\begin{lstlisting}[language=C, caption=OpenMP parallel similarity computation]
#pragma omp parallel for schedule(dynamic, 64) \
    default(none) \
    shared(local_documents, query_arr, scores, \
           local_count, dict_size, metric)
for (int i = 0; i < local_count; i++) {
    scores[i] = computeSimilarity(dict_size,
                    &local_documents[i * dict_size],
                    query_arr, metric);
}
\end{lstlisting}

    The \texttt{schedule(dynamic, 64)} clause assigns chunks of 64 documents to
    available threads at runtime.  This provides load balancing when document
    feature vectors are sparse (early exit in the inner loop), at the cost of a
    modest scheduling overhead that is negligible for the large $n$ targeted.

    \subsection{Work Distribution}

    Documents are partitioned into \textbf{contiguous blocks} using
    \texttt{MPI\_Scatterv} with per-rank displacement and count arrays
    computed on rank 0:

    \begin{equation}
        \text{sendcount}[r] =
        \left(\left\lfloor \frac{n}{P} \right\rfloor +
        [r < n \bmod P]\right) \times m
    \end{equation}

    where $[r < n \bmod P]$ is 1 if rank $r$ receives an extra document,
    0 otherwise.  A single \texttt{MPI\_Scatterv} call replaces the original
    sequential \texttt{MPI\_Send} loop, reducing scatter cost from $O(P)$
    sequential latencies to $O(\log P)$ via the MPI runtime's internal
    binomial tree \cite{thakur2005mpi}.

    \subsection{Performance Profiling}

    Each phase is timed independently using \texttt{MPI\_Wtime()}.
    Rank 0 records wall-clock timestamps before and after each phase,
    then reports absolute times and percentages of total time:

    \begin{itemize}
        \item $T_{\text{I/O}}$: rank 0 reads \texttt{documents.txt} and
              \texttt{query.txt} sequentially.
        \item $T_{\text{scatter}}$: \texttt{MPI\_Bcast} (query + metadata)
              plus \texttt{MPI\_Scatterv} (document blocks).
        \item $T_{\text{comp}}$: OpenMP-parallel similarity scoring plus
              sequential heap maintenance per rank.
        \item $T_{\text{gather}}$: \texttt{MPI\_Gather} of padded top-$k$
              arrays plus final merge sort at rank 0.
    \end{itemize}

    Additionally, the program prints the Amdahl-predicted maximum speedup
    for the measured compute fraction at the current $P \times T$
    configuration.

    \subsection{Amdahl's Law Analysis}

    From the 1-rank run, the serial fraction is:

    \begin{equation}
        f = \frac{T_{\text{I/O}}}{T_{\text{total}}} =
        \frac{0.015953}{0.027076} \approx 0.589
    \end{equation}

    Amdahl's Law predicts the theoretical maximum total speedup for any
    parallel unit count $P \cdot T$:

    \begin{equation}
        S(P \cdot T) = \frac{1}{f + \dfrac{1-f}{P \cdot T}}
    \end{equation}

    \begin{table}[h]
        \centering
        \caption{Amdahl's Law: Theoretical vs. Actual Total Speedup}
        \label{tab:amdahl}
        \begin{tabular}{|c|c|c|}
            \hline
            \textbf{Total Units ($P \times T$)} &
            \textbf{Theoretical $S(P \cdot T)$} &
            \textbf{Actual Total Speedup} \\
            \hline
            2  & 1.26$\times$ & 1.15$\times$ \\
            4  & 1.45$\times$ & 0.77$\times$ \\
            8  & 1.56$\times$ & 0.70$\times$ \\
            \hline
        \end{tabular}
    \end{table}

    Total speedup is bounded by the 58.9\% serial I/O fraction.
    Computation-phase speedup is \textit{not} bounded by Amdahl's Law
    because I/O is excluded from its denominator.

    \subsection{Discussion}

    \subsubsection{Algorithmic Improvements}
    The min-heap achieves $O(n \log k)$ top-$k$ selection
    \cite{cormen2022algorithms} with the computation phase showing near-linear
    speedup (7.13$\times$ at 8 ranks).  The two-phase OpenMP design -- separate
    parallel score computation and sequential heap insertion -- avoids locking
    while fully parallelising the expensive inner loop.

    \subsubsection{Communication Optimisation}
    Replacing $2(P-1)$ point-to-point messages with a single
    \texttt{MPI\_Scatterv} and a single \texttt{MPI\_Gather} reduces
    communication rounds from $O(P)$ to $O(\log P)$, consistent with
    the theoretical analysis of Thakur et al.\ \cite{thakur2005mpi}.

    \subsubsection{Scalability Limitations and Gustafson's Law}
    Under Amdahl's Law (strong scaling), total speedup saturates at
    approximately 1.56$\times$ due to the 58.9\% serial I/O fraction.
    However, under \textbf{Gustafson's Law} (weak scaling -- corpus grows
    proportionally with $P \cdot T$):

    \begin{equation}
        S'(P \cdot T) = P \cdot T - f \cdot (P \cdot T - 1) \approx
        (P \cdot T)(1 - f) + f
    \end{equation}

    With $f = 0.589$ and $P \cdot T = 8$: $S'(8) \approx 3.9\times$ -- far
    more favourable than Amdahl's bound.  This motivates weak-scaling
    experiments where dataset size grows with processor count.

    \section{Conclusion}

    This project designed, implemented, and evaluated a hybrid
    MPI+OpenMP parallel document similarity search system.
    The following contributions were made:

    \begin{enumerate}
        \item \textbf{Hybrid Two-Level Parallelism}: Combined MPI for
              inter-process distribution with OpenMP for intra-process
              thread-level parallelism, enabling simultaneous exploitation
              of both distributed-memory and shared-memory resources.

        \item \textbf{Min-Heap Top-K Selection}: Replaced $O(n \log n)$
              full sorting with $O(n \log k)$ heap-based selection, achieving
              near-linear computation speedup of \textbf{7.13$\times$} with
              8 MPI ranks.

        \item \textbf{Optimal Collective Communication}: Replaced sequential
              \texttt{MPI\_Send} loops with \texttt{MPI\_Scatterv} (scatter)
              and \texttt{MPI\_Gather} (result collection), reducing
              communication overhead from $O(P)$ to $O(\log P)$ latencies.

        \item \textbf{Correctness-Preserving KReduce}: Uniform sentinel
              padding to exactly $k$ entries per rank eliminates the
              \texttt{MPI\_Gather} mismatch bug present in naive implementations.

        \item \textbf{Bug Fix -- Manhattan Distance}: Replaced \texttt{abs()}
              (integer absolute value, silently truncates \texttt{double})
              with \texttt{fabs()} from \texttt{<math.h>}, restoring numerical
              correctness.

        \item \textbf{Four Similarity Metrics}: Power, Cosine, Euclidean, and
              Manhattan similarity through a unified \texttt{computeSimilarity}
              interface, enabling runtime metric selection without code
              duplication.

        \item \textbf{Four-Phase Performance Profiling}: Separate
              \texttt{MPI\_Wtime} timing for I/O, scatter, computation, and
              gather phases reveals I/O (58.9\%) as the dominant bottleneck,
              guiding future optimisation toward parallel MPI-IO.
    \end{enumerate}

    Correctness was verified across all MPI rank counts (1, 2, 4, 8) by
    confirming identical top-$k$ rankings for all similarity metrics.
    The system compiles with zero warnings under \texttt{-Wall -O3} and
    is fully automated through the provided Makefile targets.

    Future directions include: parallel MPI-IO for distributed file reading,
    GPU offload using OpenCL or CUDA for the similarity computation kernel,
    approximate search using HNSW \cite{malkov2020hnsw} for sub-linear query
    time, and weak-scaling experiments on larger corpora to validate
    Gustafson's Law predictions.
